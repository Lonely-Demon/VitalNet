# VitalNet — API Reference

A complete, current reference for every HTTP endpoint the backend exposes.
Generated by reading `backend/app/api/routes/*.py` directly — if this drifts
from the code, the code wins; please fix this doc in the same PR that
changes a route (see CODEBASE_MAP.md's "keeping documents current" rule).

For architecture/data-flow context, see `CODEBASE_MAP.md`. For the *why*
behind auth/rate-limit/CSRF design choices, see `docs/DECISIONS.md`.

---

## Conventions that apply to (almost) every endpoint

**Base URL**: `VITE_API_BASE_URL` (frontend) / wherever `uvicorn app.main:app`
is running. All paths below are relative to that.

**Auth header**: `Authorization: Bearer <supabase-jwt>` — a Supabase Auth
session access token. Verified locally (HS256, hot path) with a network
fallback for asymmetric-key projects (see `docs/DECISIONS.md` §Auth). Role
and `facility_id` are re-resolved from the `profiles` table on every request
(cached per-user for `revocation_recheck_seconds`, default 300s) — **never**
trusted from the JWT's own `user_metadata`.

**CSRF + device headers** (required on every `POST`/`PUT`/`PATCH`/`DELETE`
under `/api/*` that also carries an `Authorization` header — enforced by
`main.py::csrf_and_device_guard`):
- `X-CSRF-Token: <settings.csrf_token>` — a shared, non-secret string (the
  protection is the CORS preflight a custom header forces, not the value's
  secrecy). Must match the backend's `CSRF_TOKEN` env var exactly.
- `X-Device-Id: <any stable per-browser string>` — just needs to be
  present; used for future abuse/anomaly detection, not itself a security
  boundary.
- Missing/wrong → `403` (CSRF) or `400` (missing device id) before the route
  handler even runs.

**Rate limiting**: per-endpoint, keyed on the cryptographically **verified**
JWT `sub` (never a forgeable claim) via `slowapi`. Exceeding the limit
returns `429`. Limits are noted per-endpoint below. Storage is in-memory by
default; set `RATE_LIMIT_STORAGE_URI` (e.g. a `redis://` URL) for
multi-instance deployments so the limit is shared across processes.

**Error shape**: every error is JSON `{"detail": ...}` —a string for most
`HTTPException`s, or a list of scrubbed Pydantic validation-error dicts
(with `input`/`url`/`ctx` stripped — those can carry patient PII, see
`main.py::_scrub_validation_errors`) for `422`s. The global exception
handler never returns a raw traceback (`500` → `{"detail": "Internal Server
Error"}`, logged server-side only).

**Standard response headers** on everything: `X-Content-Type-Options:
nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`,
`Cache-Control: no-store` (API responses carry PHI — never cached),
`Permissions-Policy`, `Cross-Origin-Resource-Policy: same-site`,
`Content-Security-Policy`, `X-Request-ID` (echoes or generates a
correlation id), `Strict-Transport-Security` (non-development only).

**Role scoping** (see CODEBASE_MAP.md §5/§6 for the full model): `admin` is
always global. `doctor` is scoped to their own `facility_id` where noted.
`asha_worker` is scoped to their own submissions where noted.

## Cases (`app/api/routes/cases.py`)

### `POST /api/submit`
Submit a new patient case. Runs the ML classifier (safety-net → trained
model → NEWS2 floor) and the LLM briefing generator, then upserts.
- **Auth**: `asha_worker`, `admin`. **Rate limit**: 20/min.
- **Body**: `IntakeForm` — see `app/models/schemas.py`:
  - `patient_name` (1-100 chars), `patient_age` (0-120), `patient_sex`
    (`male`/`female`/`other`), `chief_complaint` (1-200), `complaint_duration`
    (1-50), `location` (1-200) — all required.
  - `bp_systolic` (30-300), `bp_diastolic` (10-200, must be < systolic if
    both given), `spo2` (50-100), `heart_rate` (10-250), `temperature`
    (25.0-45.0°C) — all optional.
  - `symptoms: string[]` — must be a subset of `ALLOWED_SYMPTOMS` (an
    allow-list of ~12 stable ids, e.g. `chest_pain`, `high_fever`), max 20.
  - `observations`, `known_conditions`, `current_medications` — optional
    free text, control characters stripped server-side.
  - `client_id` (UUID, optional) — idempotency key for offline-queue
    retries; resubmitting the same `client_id` upserts the same row instead
    of creating a duplicate.
  - `client_submitted_at`, `created_offline` — offline-sync metadata.
  - `human_review_requested` (bool) + `human_review_reason` (required if
    the former is true) — lets the submitter flag a case for review
    independent of the ML tier.
  - `consent_captured` (bool, **must be true or the request is rejected**),
    `consent_captured_at`.
- **Response `200`**: the created/existing `case_records` row (includes
  `triage_level`, `triage_confidence`, `risk_driver`, `id`, `created_at`,
  `facility_id`, `created_offline`).
- **Side effect**: if the result is EMERGENCY, fires a Web Push notification
  to the facility's subscribed doctors as a background task (never adds
  latency to this response).

### `GET /api/cases`
Doctor dashboard feed — cursor-paginated, sorted EMERGENCY→URGENT→ROUTINE
then newest-first within a tier.
- **Auth**: `doctor` (facility-scoped if `facility_id` set), `admin`
  (global). **Rate limit**: 60/min.
- **Query params**: `before_time`, `before_priority` (0/1/2),
  `before_id` (composite keyset cursor from the previous page), `limit`
  (default 25, capped 100).
- **Response `200`**: `{ cases: [...], hasMore, nextCursor, nextTriagePriority, nextId }`.

### `PATCH /api/cases/{case_id}/review`
Mark a case reviewed.
- **Auth**: `doctor` (own facility), `admin`. **Rate limit**: 60/min.
- **Response `200`**: `{ status: "reviewed", case_id, reviewed_by }`.

### `PATCH /api/cases/{case_id}/triage-override`
A doctor corrects the ML triage tier with a required reason — the real-label
source for retraining (see `docs/DECISIONS.md` §Retraining loop).
- **Auth**: `doctor` (own facility), `admin`. **Rate limit**: 30/min.
- **Body**: `TriageOverride` — `overridden_triage`
  (`ROUTINE`/`URGENT`/`EMERGENCY`), `override_reason` (1-500 chars).
- **Response `200`**: `{ status: "overridden", case_id, overridden_triage, overridden_by }`.

### `PATCH /api/cases/{case_id}/outcome`
Record what actually happened to the patient — the other real-label source.
Immutable: each call **inserts** a new `case_outcomes` row (corrections are
new rows, not edits).
- **Auth**: `doctor` (own facility), `admin`. **Rate limit**: 30/min.
- **Body**: `CaseOutcomeInput` — `actual_severity`
  (`ROUTINE`/`URGENT`/`EMERGENCY`), `patient_disposition`
  (`treated_discharged`/`admitted`/`referred_higher_facility`/`deceased`/`unknown`),
  `outcome_notes` (optional, ≤1000 chars).
- **Response `200`**: the created `case_outcomes` row.

### `GET /api/cases/mine`
ASHA worker's own submission history — cursor-paginated, reduced column set
(no full `briefing` JSONB — that's doctor-facing only).
- **Auth**: `asha_worker` (own submissions only), `admin`. **Rate limit**: 60/min.
- **Query params**: `before`, `before_id`, `limit` (default 25, capped 100).
- **Response `200`**: `{ cases: [...], hasMore, nextCursor, nextId }`.

### `GET /api/cases/{case_id}`
Full case detail (including the `briefing` JSONB) after row-level
authorization.
- **Auth**: `asha_worker` (own case), `doctor` (own facility), `admin`.
  **Rate limit**: 60/min.
- **Response `200`**: the full `case_records` row. `404` if not found/deleted.

## Admin (`app/api/routes/admin_routes.py`, prefix `/api/admin`)

All admin-only (`require_role('admin')`) — the **only** access boundary,
since these use the RLS-bypassing service-role client (`supabase_admin`).
`test_admin_authz.py` asserts every route here carries that guard.

### `GET /users` — 60/min
Paginated (`page`, `limit` ≤200) list of all users joined with profile data.

### `POST /users` — 10/min
Create a user + profile row. **Body** (`CreateUserRequest`): `email`,
`password` (12-128 chars, must contain upper+lower+digit+symbol —
`PASSWORD_POLICY_RE`), `full_name` (1-100), `role`
(`asha_worker`/`doctor`/`admin`), `facility_id` (required for
`asha_worker`/`doctor`), `asha_id` (optional). Rolls back the auth user if
profile provisioning fails (no orphaned accounts).

### `POST /users/bulk` — 3/min
CSV-onboarding endpoint. **Body**: `{ users: CreateUserRequest[] }` (1-100
rows). Reuses the exact same provisioning logic as `POST /users` per row —
one bad row (duplicate email, weak password) is reported without failing
the batch. **Response**: `{ results: [{row, email, status, id?|detail}], succeeded, failed }`.

### `PATCH /users/{user_id}` — 30/min
**Body** (`UpdateUserRequest`, all optional): `role`, `facility_id`,
`asha_id`, `is_active`. Updates both the `profiles` row and the JWT
`user_metadata` (rolled back together on partial failure).

### `DELETE /users/{user_id}` — 30/min
Soft-deactivate (`profiles.is_active = false`). Does not delete the auth
user or their cases.

### `POST /users/{user_id}/reactivate` — 30/min

### `GET /facilities` — 60/min
All facilities (admin view — for the doctor-accessible equivalent, see
`GET /api/facilities` under Referrals below).

### `POST /facilities` — 10/min
**Body** (`CreateFacilityRequest`): `name` (1-200), `type` (default `PHC`),
`address`, `district`, `state` (default `Tamil Nadu`), `pincode`, `phone`.

### `PATCH /facilities/{facility_id}/toggle` — 30/min
Flips `is_active` with optimistic concurrency (only succeeds if the state
hasn't changed since read — `409` on a concurrent-toggle race).

### `GET /stats` — 60/min
System-wide counts: cases by triage tier, users by role, active-user count.

### `GET /audit-log` — 60/min
Paginated (`before` cursor, `limit` ≤200) view of `phi_audit_log`.

## Analytics (`app/api/routes/analytics_routes.py`, prefix `/api/analytics`)

All `require_role('doctor', 'admin')`; `doctor` responses are scoped to
their own `facility_id`, `admin` is global (`GLOBAL_SCOPE_ROLE` constant).
Queries run concurrently with a per-query timeout — a slow/failed query
degrades that section (`_degraded: true` in the response) rather than
failing the whole endpoint.

### `GET /summary` — 60/min
Total cases, triage distribution, last-7-days daily volume, reviewed/
unreviewed counts, top-5 ASHA submitters (last 30 days).

### `GET /emergency-rate` — 60/min
EMERGENCY rate over the last 30 days, grouped by ISO week.

### `GET /response-times` — 60/min
Per-tier median/p90 review latency (last 30 days) + an overdue count
(EMERGENCY >15min, URGENT >2h, ROUTINE >24h, still unreviewed).

### `GET /ml-agreement` — 60/min
% of recorded `case_outcomes` where `actual_severity` matches the original
ML `triage_level`, overall and per-tier — the signal for "is it time to
retrain" (see `docs/DECISIONS.md` §Retraining loop).

### `GET /export` — 10/min
Streams a CSV of case records for a date range.
- **Query params** (required): `date_from`, `date_to` (ISO 8601; range
  capped at 366 days).
- **Response**: `text/csv`, `Content-Disposition: attachment`. Columns:
  `id, created_at, reviewed_at, triage_level, triage_confidence,
  overridden_triage, override_reason, risk_driver, chief_complaint,
  patient_age, patient_sex, patient_location, facility_id, submitted_by,
  reviewed_by, needs_review, triage_model_version`.
- Every export is PHI-audit-logged (bulk egress) with the row count and
  date range.

## Push notifications (`app/api/routes/push_routes.py`, prefix `/api/push`)

Returns `503` on the subscribe endpoint if `VAPID_PUBLIC_KEY`/
`VAPID_PRIVATE_KEY` aren't configured — Realtime-while-open remains the
primary channel regardless.

### `POST /subscribe` — 10/min — `doctor`, `admin`
**Body** (`PushSubscriptionInput`): `endpoint` (≤2000 chars), `p256dh_key`,
`auth_key` (≤500 chars each) — the three fields from a browser
`PushSubscription.toJSON()`.

### `DELETE /subscribe` — 10/min — `doctor`, `admin`
**Query param**: `endpoint`. Removes the caller's own subscription for that endpoint.

### `POST /check-emergency-escalations` — 6/min — `admin`
Idempotent, meant to be invoked on a schedule by an **external** scheduler
(cron / Supabase `pg_cron` / a Railway cron add-on — there is no in-process
scheduler in this backend). Re-alerts a facility's doctors for any EMERGENCY
case still unreviewed past 15 minutes, tracking `last_escalated_at` so a
case is escalated at most once per threshold window.
**Response**: `{ checked, escalated }`.

## Referrals (`app/api/routes/referral_routes.py`)

### `GET /api/facilities` — 60/min — `doctor`, `admin`
Active facilities for the referral-target picker (doctor-accessible,
unlike `/api/admin/facilities`). Excludes the caller's own facility. No
tier-based filtering — `facilities.type` is free text with no defined
ordering yet.

### `POST /api/cases/{case_id}/refer` — 20/min — `doctor` (own facility), `admin`
**Body** (`CreateReferralRequest`): `receiving_facility_id`, `reason`
(1-1000 chars), `urgency` (`ROUTINE`/`URGENT`/`EMERGENCY`). `400` if
referring a case to its own facility.

### `GET /api/referrals` — 60/min — `doctor`, `admin`
**Query param**: `direction` (`outgoing`/`incoming`/`all`, default `all`) —
ignored for `admin` (sees everything). **Response**: `{ referrals: [...] }`,
each row embedding `case_records` (chief complaint/age/sex/triage) and
`referring_facility`/`receiving_facility` names.

### `PATCH /api/referrals/{referral_id}/status` — 30/min — `doctor` (receiving facility only), `admin`
**Body**: `{ status }` — one of `acknowledged`/`patient_arrived`/
`completed`/`cancelled`. Enforces a forward-only state machine
(`pending → acknowledged → patient_arrived → completed`, `cancelled`
reachable from any active state); `409` on an invalid transition or a
concurrent-update race.

## Data-subject requests (`app/api/routes/dsr_routes.py`, prefix `/api/admin/cases`)

DPDP Act 2023 data-principal rights (`docs/COMPLIANCE_DPDP.md`). Admin-only
and scoped to a single `case_id` — VitalNet has no cross-visit patient
identifier (`docs/DECISIONS.md` §6), so locating every case for one
real-world patient is a manual admin-search step, not something these
endpoints automate.

### `GET /{case_id}/export` — 10/min — `admin`
Right to access. Returns `{ case_id, exported_at, case, outcomes,
attachments, referrals }` — every row across `case_records`,
`case_outcomes`, `case_attachments`, and `referrals` tied to that case.
Not filtered on `deleted_at` — applies regardless of soft-delete state.
Audit-logged as `PHI_EXPORT`.

### `POST /{case_id}/erase` — 10/min — `admin`
Right to erasure. Redacts `case_records`' identifying free-text fields
(`patient_name`, `patient_location`, `chief_complaint`, `observations`,
`known_conditions`, `current_medications`) and `referrals.reason` to a
fixed marker, and soft-deletes the case if not already. Vitals, symptoms,
triage output, and timestamps are preserved (de-identified clinical
signal). `case_outcomes` is never touched — it's immutable-by-design and
carries no direct identifier. **Response**: `{ status: "erased", case_id,
redacted_fields }`. Audit-logged as `PHI_ERASURE`.

### `POST /purge-expired` — 6/min — `admin`
Retention-policy sweep, meant to be invoked on a schedule by an external
scheduler — same pattern as `POST /api/push/check-emergency-escalations`.
Applies the same redaction as `/erase` to every not-yet-redacted case older
than `settings.data_retention_days`. No-op (`{ enabled: false, purged: 0 }`)
when `data_retention_days` is `0`, the default. **Response**: `{ enabled,
checked, purged }`.

## Voice transcription (`app/api/routes/voice_routes.py`, prefix `/api/voice`)

### `POST /transcribe` — 20/min — `asha_worker`, `doctor`, `admin`
Server-side transcription (Groq Whisper `whisper-large-v3`) — the accuracy
layer behind `useVoiceInput.js`'s primary voice path, with the browser's
own `SpeechRecognition` as fallback (`docs/DECISIONS.md` §15).
**Multipart body**: `file` (audio, ≤10 MB, one of
`audio/webm|wav|wave|x-wav|mp4|mpeg|ogg|m4a|x-m4a` — codec suffixes like
`;codecs=opus` are stripped before matching). **Query param**: `language`
(optional, `en`/`hi`/`ta` — anything else is ignored, letting Whisper
auto-detect). **Response**: `{ transcript }`. `415` on an unsupported
content type, `413` over the size cap, `503` if `GROQ_API_KEY` isn't
configured. No audio is persisted — transcribed and discarded within the
request.

## Security (`app/api/routes/security.py`, prefix `/api/security`)

### `DELETE /cases/{case_id}` — 30/min
Soft-delete (`deleted_at`) — reuses `cases.py`'s row-level authorization
(admin global, doctor own-facility, asha_worker own-submissions).
- **Auth**: `admin`, `doctor`, `asha_worker` (all three — authorization is
  enforced by row ownership, not just role).
- **Requires** `X-Device-Id` (part of the general CSRF/device-guard rule
  above, called out here since it's easy to miss on a DELETE).

## Health (`app/main.py`)

### `GET /api/health` — 120/min — public
- **Unauthenticated response**: `{ status: "ok"|"degraded", version }`.
- **Authenticated as `doctor`/`admin`**: adds `database` and `classifier`
  diagnostic fields (loaded/not, model version). Exception details are
  **never** included even for authenticated callers — logged server-side
  only.
- Returns `503` if the database is unreachable or the ML classifier failed
  to load (the app boots in degraded/rules-only mode rather than crashing —
  see `docs/DECISIONS.md`).

## Endpoints that are documented specs, not live code

`FEATURES_ROADMAP.md` §3 describes SMS-based submission and photo
attachments. Only the vendor-independent scaffolding exists today
(`app/services/sms.py`'s parser/gateway interface, the `case_attachments`
schema) — there is **no** `/api/sms/inbound` or `/api/cases/{id}/attachments`
endpoint live yet. Don't assume either exists without checking the roadmap
entry's current status first.
