# Classifier Evolution

| Version | File | Notes |
|---|---|---|
| v1.0 | `classifier_original.py` (deleted) | Original Phase 1, basic vitals features |
| v2.0 | `classifier_v2.py` (deleted) | Second iteration, improved recall |
| v3.0 (legacy) | `classifier.py` → `_predict_legacy()` (deleted) | 45-feature pipeline via `ClinicalFeatureEngineer` |
| v4.0 (enhanced, **retired**) | `enhanced_classifier.py` (deleted) | 4-sub-model ensemble (`emergency_detector` + `symptom_classifier` + `clinical_reasoner` + `VotingClassifier` + `CalibratedClassifierCV`). Retired because: (1) the shipped `.pkl` was empirically found to be incompatible with current scikit-learn (`ModuleNotFoundError: No module named '_loss'` on load — a live startup-crashing bug), (2) it was 25 MB and ran 3 tree models redundantly per prediction for no measurable accuracy gain over a single well-tuned model, and (3) the backend's "enhanced" model and the frontend's ONNX-exported model were trained independently on different synthetic data, so online and offline triage could disagree for the same patient. |
| v3.0.0 (unified) | `classifier.py` + `scripts/train_classifier.py` | Single `HistGradientBoostingClassifier` trained once and exported to both the backend `.pkl` and a frontend artifact — guaranteeing online and offline triage always agree. Added the deterministic safety-net override (`_safety_net_check`) escalating unambiguous critical presentations to EMERGENCY independent of the model. Evidence-informed synthetic labels (NEWS2 / qSOFA / paediatric APLS-PALS). |
| v3.0.0 (round-2 hardening) | same files + `scripts/tree_export.py`, `frontend/src/utils/{treeEvaluator,clinicalRules}.js`, `app/ml/MODEL_CARD.md` | Round-2 refinements: **(1) Offline runtime replaced** — the browser no longer loads onnxruntime-web (~12 MB WASM); the model is exported as compact `triage_trees.json` (~1 MB) and evaluated by a dependency-free JS tree walker, with a golden-vector parity test (`npm run test:parity`) asserting JS == server. **(2) NEWS2 concerning-vital floor** (`_news2_concerning_vital`) — a concerning single vital (NEWS2 score ≥ 2) can never be left ROUTINE; floors to URGENT. Mirrored in JS. **(3) Abstention** — a `low_confidence` flag (top proba < 0.55 or margin < 0.15) surfaced in the UI. **(4) Broader training** — synthetic generator now simulates missing vitals (rural no-cuff/no-pulse-ox reality) and edge syndromes (silent MI, sepsis-without-fever); 5-fold CV + ECE reporting added. **(5) Rules-only fallback** offline so triage never fails if the model can't load. Full metrics + honest limitations: `MODEL_CARD.md`. Held-out: 98.9% acc, 98.3% EMERGENCY recall (model alone), ECE 0.0016; CV: 99.2% acc.|
| v3.0.0 (round-3 ML audit) | same files + `scripts/export_golden_vectors.py`, `tests/test_feature_parity.py` | A direct audit of the trained model + SHAP explainer (docs/DECISIONS.md §23) found **43 → down from 45 features**: `time_of_day_risk` and `epidemic_alert_level` were constant across the entire 36,000-patient training set (`datetime.now()` runs once per training invocation; the latter was a hardcoded `0.0`), so the tree ensemble could never learn a split on either — removed outright. `seasonal_risk`/`geographic_risk` were rebuilt as **real signals**: `generate_patient()` now samples a `_reference_month` per synthetic patient and correlates monsoon-season (Jun–Sep) + rural/tribal location with a genuine dengue/malaria-like symptom-probability bump. Also added: a **realistic-prevalence calibration check** (~85/12/3 ROUTINE/URGENT/EMERGENCY, subsampled from the held-out test set) validating the abstention thresholds against the deployment-shape distribution, not just the balanced training/test split. **Fixed a real (if rare) parity bug** in `tree_export.py`: split thresholds were rounded to 6 decimal places, which could flip a low-cardinality discrete feature value across a split boundary in ~1-in-2000 held-out samples — increased to 9 decimal places (lossless versus the underlying float32 threshold). **Investigated and deferred:** monotonic constraints on the "higher = worse" risk-score features — `HistGradientBoostingClassifier` in the pinned scikit-learn 1.9.0 does not support `monotonic_cst` for multiclass classification (verified directly), so this was not applied without an unplanned scikit-learn upgrade. Held-out: 99.0% acc, 98.8% EMERGENCY recall (model alone), ECE 0.0020 (balanced) / 0.0050 (realistic prevalence); CV: 99.1% acc.|
| v3.1.0 (round-5 age-aware + lighter) | same files + `frontend/src/utils/treeEvaluator.js` | Round-5 ML pass (docs/DECISIONS.md §31). **(1) Fixed the model-level infant over-triage** (a documented v3.0.0 limitation): the synthetic-label scorer age-adjusted HR and temperature but scored systolic BP with adult bands, so a healthy infant's normal-low BP was labelled "hypotensive" and the label itself said EMERGENCY. Added `_pediatric_bp_score` (PALS 5th-percentile hypotension thresholds), age-gated the qSOFA hypotension criterion, made synthetic BP *generation* age-appropriate (~85+2·age for children, not the adult ~118), and added ~22% paediatric oversampling skewed to under-2s. A normal 6-month-old (HR 140, BP 85/55) now classifies URGENT (the conservative floor), not EMERGENCY; genuinely sick children still escalate. **(2) Bit-exact float32 parity** — the offline JS/reference evaluators now cast both operands to float32 (`Math.fround` / `np.float32`) before the `<=` split comparison, mirroring the server's float32 predict, so a feature value landing exactly on a split threshold can no longer take a different branch offline vs online (2/5400 held-out samples diverged before; 0 after). **(3) Lighter model** — `max_leaf_nodes=24` + learning_rate 0.09 + tighter early stopping shrank `triage_trees.json` from 1,236 KB / 747 trees to **934 KB / 702 trees (~24% smaller)** for the same accuracy; EMERGENCY class weight nudged 6→7 to protect recall. Held-out: 98.7% acc, 98.3% EMERGENCY recall (model alone), ECE 0.0031 (balanced) / 0.0087 (realistic prevalence); CV: 98.6% acc, 98.5% EMERGENCY recall; realistic-prevalence acc 99.3%.|
| **v3.1.0 (current, round-6 training-pipeline rewire — same model, new mechanism)** | `tools/training/train_classifier.py` (moved from `scripts/`) + `packages/clinical-core/cli.mjs` | Round 6 rebuild, Phase 6 (docs/DECISIONS.md §33) — **kills the last hand-mirrored pair**: `train_classifier.py` no longer carries its own ~190-line NEWS2/qSOFA/PALS Python port or its own `ClinicalFeatureEngineer` calls. It now generates synthetic patients as before, then pipes them through `packages/clinical-core/cli.mjs` (a JSONL subprocess) for both labeling (`assignTier`, the same rules engine ported FROM this file's v3.1.0 scorer and verified 100.000% conformant against it — see §33's conformance report) and feature engineering (`buildFeatureMap`) — in `CLI_BATCH_SIZE`-sized batches (8,000 patients/call) to keep subprocess overhead low during the per-class rejection-sampling loop. **No model-behaviour change**: this is a training-infrastructure change, not a retrain — labels and features are computed by the exact rules that were ported from this scorer, so a full production retrain is expected to reproduce v3.1.0's numbers (not independently re-verified at full 36,000-patient scale in this change; the CLI wiring itself was verified via `tools/training/smoke_test.py`, CI-enforced, and a live end-to-end run of `retrain_from_outcomes.py`/`fairness_audit.py` against their existing fixtures). `retrain_from_outcomes.py` and `fairness_audit.py` were updated in the same change to route through the same batched CLI functions instead of their own direct `ClinicalFeatureEngineer`/`assign_triage_label` calls, so nothing downstream still depends on the deleted Python scorer. Committed model artifacts are unchanged by this entry — still the round-5 `.pkl`/`triage_trees.json` above. |

## Regenerating the model

```bash
pnpm --filter @vitalnet/clinical-core build
cd tools/training
source ../../backend/venv/bin/activate
pip install -r ../../backend/requirements.txt -r ../../backend/requirements-train.txt
python train_classifier.py
```

This is the **only** supported training entrypoint. Labels and features both come from `packages/clinical-core` (via `cli.mjs`) — there is no Python scoring/feature-engineering copy left to keep in sync. `colab/triage_classifiers.py` is a historical Google Colab reference script trained on only the 14 basic (non-engineered) features — it predates `ClinicalFeatureEngineer` and is not wired into the app; keep it only for historical reference, do not use its output as a production model.
